# Jira 버그 티켓 자동 생성

사용자가 보고한 버그/에러를 버그 전용 포맷으로 정리한 뒤, 확인을 거쳐 Jira(Cloud)에 티켓을 생성한다.

## 입력

사용자가 전달한 설명: `$ARGUMENTS`

## 절차

다음 단계를 **순서대로** 따른다. 절대 확인 없이 티켓을 생성하지 않는다.

### 1단계 — 버그 내용 파악

- `$ARGUMENTS` 에 내용이 **있으면**: 그 설명을 버그 내용으로 사용한다.
- `$ARGUMENTS` 가 **비어 있으면**: 지금까지의 대화 맥락과 최근 코드 변경/에러 로그(필요하면 `git diff`, `git log --oneline -5` 확인)를 근거로 어떤 버그인지 추론한다. 맥락이 불충분하면 임의로 만들지 말고 사용자에게 무엇에 대한 버그인지 되묻는다.

### 2단계 — 버그 포맷에 맞춰 초안 작성

아래 포맷으로 **제목**과 **본문**을 작성한다.

- 제목 접두사는 `[버그]` 로 한다.
- 본문은 Jira 위키 마크업 문자열로 그대로 들어가므로, 아래 구조를 유지한다.
- 알 수 없는 항목은 추측해 채우지 말고 비워 두거나 "확인 필요"로 남긴다. 모르면 사용자에게 질문한다.
- **재현율**과 **영향 범위**는 체크박스 목록이다. 내용에 맞는 항목 **하나**의 `[ ]` 를 `[x]` 로 바꾸고 나머지는 그대로 둔다. 판단 근거가 없으면 모두 `[ ]` 로 두고 사용자에게 확인한다.

```
제목: [버그] 한 줄 요약

h2. 발생 환경
* OS: 
* 브라우저/앱 버전: 
* 디바이스: 
* 서비스 버전: 

h2. 발생 위치
* 메뉴/페이지: 
* URL: 

h2. 이슈 내용
무엇이 어떻게 문제인지 구체적으로

h2. 정상 동작 기준
기대하는 정상 동작을 구체적으로

h2. 재현 절차
# 
# 
# 

h2. 재현율
[ ] 항상 발생: 10회 중 9~10회(≥90%)
[ ] 자주 발생: 10회 중 5~8회(50~80%)
[ ] 간헐적: 10회 중 2~4회(20~40%)
[ ] 드물게 발생: 10회 중 1회(≈10%)
[ ] 1회: 1번만 발생, 재현 불가
시도/발생 횟수: 

h2. 영향 범위
[ ] 전체 사용자
[ ] 특정 환경

h2. 에러 로그 / 증빙
* 콘솔 로그: 
* 네트워크 응답: 
* 스크린샷: 

h2. 추가 정보
* 관련 이슈: 
```

작성한 **제목과 본문 전체를 사용자에게 보여주고**, 함께 아래 옵션을 묻는다(전부 생략 가능):

> "이대로 생성할까요? 아래 옵션도 지정할 수 있어요 (생략 시 기본값):
> - 우선순위: Highest / High / **Medium(기본)** / Low / Lowest
> - 담당자: 이메일/이름 (생략 시 **나에게 할당**)
> - 레이블: 쉼표로 구분 (생략 시 없음)"

사용자가 자연어로 답하면 해당 값만 4단계에서 플래그로 전달한다.

### 3단계 — 사용자 확인

- 사용자가 수정을 요청하면 반영 후 다시 2단계처럼 보여주고 재확인한다.
- 사용자가 거부하면 티켓을 생성하지 않는다.
- 사용자가 승인하면 4단계로 진행한다.

### 4단계 — 티켓 생성

1. 확정된 **본문**을 임시 파일에 저장한다. 예: `Write` 로 `/tmp/jira-bug-desc.md` 에 본문 작성.
2. 플러그인에 포함된 스크립트를 실행한다 (`${CLAUDE_PLUGIN_ROOT}` 는 플러그인 루트로 치환됨). 사용자가 지정한 옵션만 플래그로 덧붙인다(미지정 플래그는 생략 — 스크립트가 기본값 처리):

   ```bash
   "${CLAUDE_PLUGIN_ROOT}/scripts/create-jira-ticket.sh" \
     --summary "확정된 제목" \
     --description-file /tmp/jira-bug-desc.md \
     --priority "High" \
     --assignee "hong@example.com" \
     --labels "bug,login"
   ```

   - `--priority`: 사용자가 안 정했으면 **플래그 자체를 생략**한다(스크립트 기본값 Medium).
   - `--assignee`: 안 정했으면 생략(스크립트가 나에게 할당). 검색 결과가 여러 명이면 스크립트가 후보를 알려주니, 사용자에게 다시 물어 이메일로 재지정한다.
   - `--labels`: 안 정했으면 생략.

3. 스크립트 출력 결과를 사용자에게 그대로 전달한다.
   - 성공: 생성된 티켓 키와 URL 안내.
   - 실패(환경변수 누락·인증·권한 등): 스크립트가 알려준 원인과 해결 방법을 안내한다. 환경변수 미설정이면 `JIRA_EMAIL`, `JIRA_API_TOKEN` 설정 방법을 알려준다.

## 주의

- 재현 절차·실제 동작을 임의로 지어내지 않는다. 근거가 있는 내용만 적고, 모르면 "확인 필요"로 남긴다.
- 사용자 승인 전에는 어떤 경우에도 스크립트를 실행하지 않는다.
